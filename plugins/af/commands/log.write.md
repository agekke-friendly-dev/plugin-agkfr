# セッションログ作成 [git,lite,detail,update]

セッション終了時に引継ぎ用のログファイルを作成します。

## 引数
- $ARGUMENTS: `git` | `lite` | `detail` | `update` | なし（複数指定可）
  - `git`: 全リポジトリ状態チェック & 同期
  - `lite`: **中間保存モード** - ログファイル作成のみ（同期・push・Notionスキップ）
  - `detail`: **詳細モード** - 通常より詳細なログを作成（実施内容の粒度を細かく、判断理由・試行錯誤・ユーザーとのやり取りの経緯も記録）
  - `update`: 直近のログに作業内容を追記
  - なし: 通常モード

**組み合わせ:**
- `detail git` → 詳細ログ + 全リポジトリチェック
- ~~`detail lite`~~ → **不可**（detailとliteは排他）

**例:**
- `/c:log.write git` → ログ作成 + 全リポジトリチェック & 同期
- `/c:log.write lite` → ログファイル作成のみ（中間保存）
- `/c:log.write detail` → 詳細ログ作成 + 同期処理
- `/c:log.write` → ログ作成 + 同期処理

**自動処理:** `docs/` フォルダが存在する場合は自動的に更新

## 手順

### 1. モード判定

| 引数 | モード |
|------|--------|
| `update` | **追記モード** → Step 1a へ |
| `lite` | **中間保存モード** → Step 2-5 → Step 6c → Step 8 → **Step 10（タイムスタンプ表示を忘れないこと！）** → 終了（他の同期・pushスキップ） |
| `detail` | **詳細モード** → Step 2 へ（ログ作成時にStep 5の詳細ルールを適用）。同期処理は通常通り実行 |
| その他 | **新規作成モード** → Step 2 へ |

**排他チェック:** `detail` と `lite` が同時指定された場合はエラーとし、「detailとliteは同時に指定できません」と報告して終了。

---

### Step 1a: 追記モード（update）

#### 1a-1. 直近のログファイル検索

```bash
ls -t for_claude/session_logs/*.md | head -1
```

最新の `.md` ファイルを取得。

#### 1a-2. 追記内容の整理

このセッションで行った**追加作業**を整理:
- 追加で完了したタスク
- 新たに発生した問題と解決
- 更新された残タスク・注意点

#### 1a-3. ログファイルに追記

ファイル末尾に以下を追加:

```markdown

---

## 追記: YYYY-MM-DD HH:MM

### 追加作業
- [x] タスク1
- [x] タスク2

### 補足・変更点
- ...

### 更新された残タスク
- [ ] ...
```

#### 1a-4. 完了報告 → Step 6 へ → ... → **Step 10**

---

### Step 2: セッション分析（新規作成モード）

このセッションで行った作業を振り返り、以下を整理:
- 完了したタスク
- 発生した問題と解決方法
- 残タスク・未完了事項
- 次回への注意点
- **お客様への連絡事項**（伝えること、質問すること）

### 3. [main]タグ判定

**[main] をつける（プロジェクト開発あり）:**
- ソースコード（.py, .js, .vba, .bas 等）を編集した
- 新しいファイルを作成した
- バグ修正・機能追加を行った
- 設計上の決定をした
- テストを実行して問題を解決した

**タグなし（軽微な作業）:**
- セッション開始/終了処理のみ
- 既存情報の確認・引継ぎのみ
- 環境設定・TODO作成のみ
- ドキュメント閲覧のみ

**迷ったら [main] をつける**（見逃すより安全）

### 4. ファイル名決定

```
# [main] の場合
YYYYMMDD_HHMMSS_[main]_作業内容.md

# タグなしの場合
YYYYMMDD_作業内容.md
```

作業内容は簡潔に（例: `VBA修正`, `仕様書作成`, `バグ修正`）
**注意:** `log_` プレフィックスは付けない

### 5. ログファイルの作成

以下の形式でMarkdownファイルを作成:

```markdown
# セッションログ: [タイトル]

作成日時: YYYY-MM-DD HH:MM

## セッション概要
[今回の作業内容を簡潔に]

## 実施内容（時系列）
1. [最初にやったこと]
2. [次にやったこと]
3. [その後やったこと]
...

## 完了タスク
- [x] タスク1
- [x] タスク2

## 発生した問題・解決方法
- 問題: ...
- 解決: ...

## ユーザーへの注意
> **重要:** [忘れないでほしいこと、覚えておいてほしいこと]
>
> - [具体的な注意点1]
> - [具体的な注意点2]

## 次回への引継ぎ
### 残タスク
- [ ] ...

### 注意点
- ...

### 推奨する次のアクション
1. ...

### よく使うスクリプト
| スクリプト | 用途 |
|-----------|------|
| `python for_claude/scripts/xxx.py` | 説明 |
```

#### detailモードの場合の追加ルール

`detail`モードでは、通常モードのテンプレートに加え、以下の粒度で記録する:

- **実施内容**: 各作業ステップをさらに細分化し、判断理由・試行錯誤・ユーザーとのやり取りの経緯も記録する
- **問題・解決**: ミスをした場合はその原因分析（なぜそうしたか）も記録する
- **変更ファイル一覧**: セッション中に変更した全ファイルのパスを列挙する
- **ユーザーからのフィードバック**: セッション中にユーザーから受けた指摘・修正指示を記録する
- 会話の流れを追えるレベルの詳細度を目指す（ただしコードの行番号は不要）

**「ユーザーへの注意」セクションの記載基準:**
- 今回のセッションで重要な決定をした場合
- 忘れると問題が起きる設定・変更をした場合
- 次回以降も意識すべきパターンや制約がある場合
- 該当なしの場合は「特になし」と記載

### 6. セッション終了時の同期処理

**※ `lite` モードの場合は Step 2-5（ログ作成）のみ実行し、他はスキップ → Step 8 → Step 10（タイムスタンプ表示）へ。Step 10 は必須。**

以下の処理を順に実行:

#### 6-1. __memory.md 同期

cwd に `__memory.md` がある場合、以下を更新:

1. **完了タスク削除**: TODOセクションの完了タスク（`- [x]`）を削除
2. **残タスク追記**: 新たな残タスクがあれば追記
3. **お客様への連絡事項追記**: セッション中に「お客様に伝えるべきこと」「お客様に確認すべきこと」があれば追記

**「お客様への連絡事項」セクションのフォーマット:**

```markdown
## お客様への連絡事項

### 伝えること
- [ ] [伝える内容] (YYYYMMDD)

### 質問すること
- [ ] [質問内容] (YYYYMMDD)
```

**追記基準:**
- 仕様確認が必要な事項
- 進捗報告が必要な事項
- 判断を仰ぐ必要がある事項
- バグ報告・問題報告
- 該当なしの場合は追記しない

**注意:** 完了済み（`- [x]`）は削除しない。ユーザーが手動で削除するまで残す。

#### 6-2. TODO.md 同期

cwd に `TODO.md` がある場合、以下を更新:

1. **完了タスク更新**: セッション中に完了したタスクを `- [x]` に変更
2. **残タスク追記**: 新たな残タスクがあれば適切な優先度セクションに追記
3. **不要タスク削除**: 完了済みかつ不要なタスクは削除

---

### 6b. CLAUDE_LIBRARY 追記（常に実行）

セッション中に学んだ技術知見があれば積極的にCLAUDE_LIBRARYへ追記する。

**追記すべき内容:**
- 新しく発見したエラーの原因と解決方法
- API/ライブラリの使い方で学んだこと
- ハマりやすいポイントと回避策
- 便利なコードパターン

**追記しない内容:**
- 単なる作業記録（ログに残す）
- プロジェクト固有の情報
- 既に記載済みの内容

#### 6b-1. カテゴリ判定

セッション内容から該当する技術カテゴリを判定:

| キーワード | ファイル |
|------------|----------|
| VBA, Excel, マクロ, .bas, .xlsm, xlsx, openpyxl, xlrd, xlwt, win32com, pywin32 | VBA.md |
| Python, pip, venv, .py | Python.md |
| PowerShell, ps1 | PowerShell.md |
| Git, commit, push, pull | Git.md |
| エンコーディング, UTF-8, CP932, BOM | Encoding.md |
| Claude Code, hooks, settings.json | ClaudeCode.md |
| Google API, Drive, OAuth | GoogleAPI.md |

**重要:** PythonスクリプトでもExcel操作（openpyxl, win32com, pywin32）はVBA.mdに分類。

複数カテゴリに該当する場合は複数ファイルを対象とする。

#### 6b-2. 既存内容確認

```bash
cat "${CLAUDE_PLUGIN_ROOT}/library/CLAUDE_LIBRARY/[カテゴリ].md"
```

該当カテゴリファイルを読み込み、既に同様の内容があるか確認。

#### 6b-3. 追記判定

以下の場合のみ追記:
- 新しい知見・ノウハウがある
- 既存の情報に追加・補足が必要
- トラブルシューティングで有用な情報

**既に記載済み or 価値なしの場合はスキップして報告**

#### 6b-4. 追記フォーマット

```markdown
## YYYY-MM-DD: [見出し - 内容が推測できる完全な文]

[内容を簡潔に]

```

**ルール:**
- 見出し = 内容が推測できる完全な文
- 本文 = 要点のみ簡潔に
- `---` の後に追記

---

### 6c. プロジェクトドキュメント更新

#### **[必須] CLAUDE.md（全プロジェクト）**

**⚠️ 忘れがち: 必ずCLAUDE.mdをReadして確認すること**

cwd に `CLAUDE.md` がある場合、セッション中の変更を反映:
- **新規ファイル/フォルダ作成 → 関連ファイルテーブルに追記**（最も忘れやすい）
- 重要な設定変更 → 該当セクションを更新
- 変更なし → スキップ

#### CLAUDE_SPECS.md（存在する場合のみ）

cwd に `CLAUDE_SPECS.md` がある場合のみ更新:
- 関数を追加/変更した → 主要関数セクションに追記
- 処理フロー変更 → 該当セクションを更新
- 変更なし → スキップ

**注意:** CLAUDE_SPECS.md がない場合は作成しない（既存ファイルの更新のみ）

#### CLAUDE_IMPORTANT.md（存在する場合のみ）

cwd に `CLAUDE_IMPORTANT.md` がある場合、セッション中の変更を反映:
- 重要なルール・制約の追加/変更
- お客様確認待ち事項の更新（回答受領/新規質問追加）
- 最優先タスクの変更
- 致命的な注意事項の追加
- 変更なし → スキップ

#### docs/（フォルダ存在時は常に更新）

cwd に `docs/` フォルダがある場合は自動的に更新:

**1. plans/PROGRESS.md 更新（存在する場合）:**
- 完了タスク → 該当行を `[x]` に変更
- 新規タスク → 追記
- 進捗率・日付を更新

**2. specs/ 内の仕様書更新:**
- セッション中に仕様変更 → 該当ファイルを更新
- 新規仕様書作成済み → 確認のみ

**3. 仕様書/ 更新（存在する場合）:**
- __memory.md の内容を反映
- セッション中の変更を各mdに反映

**更新対象の判断:**
- セッション中に関連ファイルを編集した → 更新
- 変更なし → スキップ

作成/更新したファイルパスを報告（確認は不要）

### 7b. 後処理スクリプト実行

settings.jsonバックアップ + requirements.txt更新を一括実行:

python ${CLAUDE_PLUGIN_ROOT}/scripts/log_post_process.py

### 8. 次やることを表示

セッション分析で整理した内容を元に、**次回やるべきこと**を箇条書きで表示:

```markdown
## 次やること

1. [最優先タスク]
2. [次に重要なタスク]
3. ...
```

- 残タスクから優先度の高いものを抽出
- 推奨する次のアクションを具体的に記載
- ユーザーがすぐ作業を再開できるようにする

### 9. 全リポジトリチェック（git引数がある場合のみ）

引数に `git` が含まれる場合、最後に全リポジトリの状態をチェック:

```bash
python ${CLAUDE_PLUGIN_ROOT}/scripts/git_check.py
```

**`[OK]` 以外のリポジトリがあれば即座に同期する（確認不要）:**
```bash
python ${CLAUDE_PLUGIN_ROOT}/scripts/git_check.py --sync -y
```

### 9b. employee-reports 機能（配布版では削除）

配布版では employee-reports skill が含まれないため、本セクションは除外されています。個人運用版でのみ利用可能。

### 10. セッション情報表示（常に最後に実行）

**全引数・全モード共通で、全処理の最後に以下を表示する:**

```
p:log.write $ARGUMENTS yyyy/mm/dd hh:mm:ss [プロジェクト名]
```

- 日時は処理完了時点の現在日時
- `$ARGUMENTS` はユーザーが送信した引数をそのまま表示（なしの場合は省略）
- プロジェクト名はcwdのフォルダ名をそのまま表示
